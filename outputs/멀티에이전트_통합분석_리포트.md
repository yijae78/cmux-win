## 1. Executive Summary

신교수님의 디지털 작업 환경은 **세 개의 폴더가 하나의 살아 있는 시스템**으로 연결되어 있습니다. 각 폴더의 역할을 한 줄로 요약하면 다음과 같습니다.

- `C:\projects\sermon` — 설교 콘텐츠 자산의 **저장소(Storage Layer)**. Markdown을 단일 진실 원본(Single Source of Truth)으로 두고 PDF·PPTX·HWPX 등 다매체로 자동 변환되는 데이터 레이어입니다.
- `C:\dev\cmux-win\src` — 멀티에이전트 오케스트레이션의 **실행 엔진(Runtime Engine)**. Electron Main Process가 SOT 스토어 + 5단계 미들웨어 + 10개 소켓 핸들러로 Claude·Gemini·Codex를 동시에 부려서 sermon 같은 콘텐츠를 처리합니다.
- `C:\dev\cmux-win\docs` — 위 두 폴더가 어떻게 만들어졌는지 기록한 **설계·성찰 아카이브(Design & Reflection Archive)**. 22개 문서가 v1→v3 진화 과정과 4차에 걸친 성찰 패치를 추적합니다.

세 폴더의 관계는 **"docs(설계) → src(구현) → sermon(활용)"** 의 인과 사슬을 이루며, sermon 폴더 안의 `.claude/skills/`(image-selector, ppt-maker 등)는 src의 에이전트 인프라가 직접 호출하도록 설계되어 있습니다(docs/experiment-plan.md Phase 4.3에서 이 연결을 이미 검증).

이번 분석에서 발견된 **가장 중요한 인사이트**는, 신교수님의 워크플로우가 이미 "콘텐츠–인프라–문서"라는 3계층 자족 구조를 형성하고 있다는 점이며, 다음 단계의 가장 큰 갭은 **사용자 매뉴얼·Socket API 카탈로그·Cowork Bridge 가이드 같은 운영 문서의 부재**입니다(7번 섹션 참조).

Gemini가 보고한 분포는 다음과 같으며, 각 항목 옆 괄호는 Gemini의 어림 추정치입니다.

### 2.3 설교 콘텐츠 자산의 분류 체계

Gemini는 4계층 분류를 식별했습니다.

1. **Raw Data** (`mydata/`, `mydata2/`) — 과거 설교 PDF와 거기서 추출한 이미지·텍스트 자료
2. **Archive** (`myppt/`) — 과거 사용 완성본 PPT 파일들
3. **Active Projects** (`sermons/`) — `YYYYMMDD-제목` 형식 폴더, 한 설교에 필요한 모든 소스(원고·이미지·출력물)를 독립 공간에 관리
4. **System Assets** (`templates/`, `scripts/`, `references/`) — 작성을 돕는 템플릿·자동화 도구·스타일 가이드라인

### 2.4 워크플로우 관점에서 주목할 만한 패턴

- **Markdown 기반 Single Source 워크플로우** — 모든 설교는 `sermons/` 내 `.md`로 먼저 작성되고, `기본원고.md`, `요약.md` 등 버전별로 관리됩니다.
- **자동화된 다매체 변환** — `scripts/md_to_docx.py`, `md_to_pptx.py`, `md_to_hwpx.py` 등을 통해 Markdown 원고를 교회 행정·예배 발표용 포맷으로 자동 변환합니다.
- **AI/에이전트 활용** — `.claude/skills/`에 `sermon-writer`, `ppt-maker`, `image-selector` 스킬이 정의되어 AI가 작성·시각자료 생성을 보조하는 에이전틱 워크플로우가 이미 구축되어 있습니다.
- **시각화 기획 프로세스** — 개별 설교 폴더 안에 `visual_brief.md` + `images/`가 별도로 존재하여, 설교문 작성 → 시각화 기획 → 이미지 선정의 단계가 분리되어 있습니다.

### 3.4 코드 품질 — 강점과 개선 포인트

**강점**:

- **Strict 타입 + 런타임 검증 이중 게이트** — TS 5.4 strict 전체 ON + Zod 양방향 검증(IPC, 소켓, Action payload 모두)
- **명확한 SOT 분리** — Renderer는 읽기 전용 구독자, 모든 변경은 Action Dispatch 단일 경로(설계안 §2.2)
- **5단계 미들웨어 체인** — Validation→Mutation→SideEffects→Persistence→IpcBroadcast→AuditLog로 책임 분리(audit 추적 가능)
- **에러 격리** — `process.on('uncaughtException')`이 ConPTY `AttachConsole` 에러 같은 알려진 비치명 예외만 흡수, 그 외는 그대로 전파(에러 삼키지 않음)
- **Single-instance lock** — 텔레그램 봇 409 충돌 + PTY/소켓 자원 경합 방지
- **Plan B 폴백 다수** — Claude Teams 미감지 시 자체 오케스트레이션 모드, ConPTY 실패 시 winpty 폴백 등(설계안 §11)

**개선 포인트**:

- **Codex 자동승인 미흡** — 이번 분석에서 Codex가 `(Get-Content ... | Measure-Object -Line).Lines` 같은 PowerShell 파이프라인 명령에 대한 권한 승인 루프에 갇힘. Auto-Approver 패턴에 Codex의 "Yes, proceed (y) / Press enter to confirm" UI 매칭을 강화해야 합니다.
- **알려진 9개 실패 테스트** — 전체 416개 중 tmux-shim 3 + history-db 6 실패(2026-03-24 retrospective 기준). 회귀는 없지만 누적 부채.
- **PowerShell 셸 통합 로딩** — F23(설계안 4차 성찰)에서 `-NoExit -Command ". 'path'"` 인자 추가가 명시되었으나 PTY spawn 경로별 일관성 추가 검증 필요.
- **Codex 패널 응답 정체** — 이번 세션에서 Codex 패널은 Working 인디케이터만 반복 출력. Codex `--full-auto --no-alt-screen` 모드의 ink TUI 라이브 버퍼 폴링 안정성 점검 권장.

## 5. 교차 인사이트 — 세 영역은 어떻게 연결되는가

세 폴더는 **3계층 자족 시스템**을 형성합니다.

**연결 1: docs → src 인과 관계**
docs의 4차에 걸친 성찰(F1~F25)이 src의 코드 베이스에 직접 반영되어 있습니다. 예) F2(stdin JSON에서 session_id 읽기) → `src/cli/claude-hook-session-store.ts`, F10(자체 오케스트레이션) → `src/main/socket/handlers/agent.ts`(273행), R2(소켓 인증) → `src/main/socket/auth.ts`. 곧 **docs는 src의 변경 이력을 동시에 추적하는 살아 있는 RFC 모음**입니다.

**연결 2: src ↔ sermon 인프라-콘텐츠 결합**
sermon의 `.claude/skills/`에 정의된 `sermon-writer`, `ppt-maker`, `image-selector` 스킬은 cmux-win의 에이전트 오케스트레이션 인프라가 직접 호출하도록 설계되어 있습니다. docs/experiment-plan.md Phase 4.3에서 이미 "Gemini가 sermons/ 6개 폴더 정확히 보고"로 검증되었고, 이번 분석에서 다시 동일 패턴(Gemini → C:\projects\sermon 분석)이 정상 동작함을 재확인했습니다.

**연결 3: docs ↔ sermon 워크플로우 거울 관계**
docs의 retrospective(`2026-03-24-full-day-retrospective.md`)와 sermon의 `references/에이전틱 워크플로우 가이드`는 동일한 사고 모델("리더-워커, 자동승인, Markdown SOT")을 거울처럼 공유합니다. 신교수님이 cmux-win 개발에서 얻은 교훈이 sermon 콘텐츠 워크플로우에 그대로 이식되어 있습니다.

**갭(gap) 식별**:

- **Sermon 콘텐츠가 cmux의 워크플로우 자동화 기능을 풀로 활용 못 하고 있음** — `workflow.run` RPC(experiment Phase 5.3)는 3프로젝트 순차 실행에 성공했지만, sermon의 일상 작성 루틴(MD 작성→DOCX→PPTX→HWPX)이 workflow로 정형화되어 있지 않습니다.
- **사용자 매뉴얼·Socket API·Bridge 사용법** 문서가 부재하여, 신교수님이 핸드폰에서 텔레그램으로 설교 작성을 원격 지시할 때마다 매번 명령어를 회상해야 합니다.
- **Sermon 폴더에 자체 README/INDEX 부재** — Gemini가 보고한 11개 1차 폴더의 역할 분담이 명문화된 문서가 sermon/ 안에 없음(추측).
