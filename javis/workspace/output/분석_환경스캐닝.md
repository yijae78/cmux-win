## 1. 4중 워크플로우 구조

### 1.1 공통 구조

시스템은 WF1~WF4를 독립 워크플로우로 분리하고, 각 워크플로우가 동일한 데이터 단계 `raw/structured/filtered/analysis/signals/reports/`를 갖도록 설계되어 있다. 문서상 핵심 원칙은 워크플로우 독립성이다. WF1은 WF2~WF4의 존재를 모르고, WF2는 WF1/WF3/WF4를 모르며, WF3와 WF4도 같은 방식으로 상호 독립이다. 통합은 개별 워크플로우 내부가 아니라 병합 및 대시보드 계층에서 수행된다.

각 워크플로우의 공통 실행 흐름은 대략 다음과 같다.
1. 소스 설정 로드 및 수집
2. 원천 데이터 저장과 실행 증명 기록
3. 구조화 및 중복 제거
4. STEEPs/약신호 분석
5. 시그널 데이터베이스와 스냅샷 갱신
6. 개별 보고서 생성
7. 통합 보고서와 대시보드 병합

### 1.2 WF1: General

역할: 일반 환경스캐닝 소스에서 광범위한 사회, 기술, 경제, 환경, 정치 신호를 수집하는 기본 워크플로우다. 문서상 WF1은 arXiv를 제외하도록 분리되어 있으며, SOT 규칙에서 WF1 소스의 arXiv `enabled: false`가 강제된다.

주요 소스와 경로:
- 소스 설정: `env-scanning/config/sources.yaml`
- 데이터 루트: `env-scanning/wf1-general/`
- 오케스트레이터: `env-scan-orchestrator` 계열 에이전트 또는 커맨드
- 산출물: `wf1-general/raw`, `structured`, `filtered`, `analysis`, `signals`, `reports`

특징: WF1 영향도 점수는 v3.6.0 변경사항에서 0~100 스케일을 0~10으로 정규화한 것으로 문서화되어 있다. 이 변경은 WF 간 통합 점수 비교 가능성을 높이지만, 과거 데이터와 혼합될 때 정규화 여부 메타데이터가 없으면 해석 오류가 생길 수 있다.

### 1.3 WF2: arXiv

역할: 학술 논문 기반 약신호를 별도 채널로 수집한다. WF2는 arXiv 전용 소스를 활성화하며, 일반 웹/뉴스 소스와 분리해 학술 신호의 특성을 보존한다.

주요 소스와 경로:
- 소스 설정: `env-scanning/config/sources-arxiv.yaml`
- 데이터 루트: `env-scanning/wf2-arxiv/`
- 실행 명령: `/env-scan:run-arxiv`
- 검증: Pipeline Gate 2에서 `validate_phase2_output.py` 강제 수행으로 문서화되어 있다.

평가: 학술 소스는 노이즈가 낮고 원천 신뢰도가 높지만, 사회 확산 전 단계의 신호가 많아 영향도 산정에서 뉴스/소셜 소스와 동일 가중치를 적용하면 과대 또는 과소 평가될 수 있다. WF2를 독립시킨 설계는 이 문제를 완화한다.

### 1.4 WF3: Naver

역할: 한국어 네이버 뉴스 기반 국내 신호를 수집하는 워크플로우다. 글로벌 영어권 신호와 다른 한국 로컬 이슈, 정책, 소비자 반응, 산업 동향을 포착하는 데 초점이 있다.

주요 소스와 경로:
- 소스 설정: `env-scanning/config/sources-naver.yaml`
- 데이터 루트: `env-scanning/wf3-naver/`
- 실행 명령: `/env-scan:run-naver`
- 관련 모듈: 문서상 `naver_crawler.py`는 `NaverNewsCrawler`와 `CrawlDefender`, `naver_signal_processor.py`는 FSSF, Three Horizons, Tipping Point, Anomaly 처리를 담당한다.

평가: WF3는 언어와 플랫폼 특성이 강하므로 중복 제거와 키워드 매칭이 영어 중심 규칙에 종속되면 누락 가능성이 크다. 한국어 형태소 처리, 동의어 사전, 기관명 정규화가 품질의 핵심이다.

### 1.5 WF4: Multiglobal News

역할: 다국어 글로벌 뉴스 소스에서 국가와 언어권을 넘나드는 약신호를 수집한다. 문서상 `sources-multiglobal-news.yaml`은 43개 사이트와 11개 언어를 다루며, WF4는 FSSF, Three Horizons, Tipping Point, 다국어 처리를 포함한다.

주요 소스와 경로:
- 소스 설정: `env-scanning/config/sources-multiglobal-news.yaml`
- 데이터 루트: `env-scanning/wf4-multiglobal-news/`
- 실행 명령: `/env-scan:run-multiglobal-news`
- 검증 프로필: `workflows.wf4-multiglobal-news.validate_profile = multiglobal-news`

평가: WF4는 가장 가치가 큰 동시에 가장 위험한 워크플로우다. 언어별 번역 품질, 지역별 매체 편향, 중복 기사 신디케이션, 시간대 정규화, 동일 사건의 다국어 표현 통합이 모두 필요하다. 문서상 Cross-WF Jaccard 강화와 공유 키워드 기반 제한이 있어 중복과 보강 신호를 분리하려는 의도는 명확하다.

## 2. 약신호 탐지 및 분류 로직

### 2.1 STEEPs 분류

문서상 STEEPs 정의는 `env-scanning/config/domains.yaml` 및 `.claude/skills/env-scanner/references/steep-framework.md`에 위치한다. 분류 축은 Social, Technological, Economic, Environmental, Political로 이해되며, 시스템은 도메인 키워드와 신호 분석 결과를 이용해 수집 항목을 STEEPs 카테고리로 배정한다.

강점:
- 환경스캐닝 표준 프레임워크를 설정 파일로 분리해 도메인 확장이 가능하다.
- WF별 수집 채널이 달라도 동일 STEEPs 축으로 통합 가능하다.
- 대시보드에서 STEEPs x Impact 형태의 시그널 맵을 제공하도록 설계되어 해석 가능성이 높다.

리스크:
- 키워드 기반 규칙이 중심이면 다의어, 은유 표현, 한국어/다국어 번역 표현에서 오분류가 발생한다.
- 하나의 신호가 여러 STEEPs에 걸칠 때 단일 라벨만 허용하면 복합 신호의 의미가 손실된다.
- 문서만으로는 분류 confidence, 다중 라벨 정책, 사람이 조정한 라벨의 재학습 여부를 확인할 수 없다.

### 2.2 5-Layer 약신호 처리 체계

문서에는 VEV 사후 검증의 3-Layer가 명시되어 있고, WF3/WF4 신호 처리에는 FSSF, Three Horizons, Tipping Point, Anomaly가 언급된다. 이를 약신호 탐지 관점에서 재구성하면 다음 5개 분석 레이어로 동작하는 설계다.

1. 수집 레이어: WF별 소스에서 raw 데이터를 수집하고 `scan_metadata.execution_proof`로 실행 증명을 남긴다.
2. 정제 레이어: structured/filtered 단계에서 구조화, 중복 제거, 기간별 archive loader 적용을 수행한다.
3. 도메인 분류 레이어: STEEPs 키워드와 도메인 설정으로 신호 후보를 분류한다.
4. 신호 평가 레이어: FSSF, Three Horizons, Tipping Point, Anomaly 분석으로 약신호의 성숙도, 시간 지평, 임계점 접근성, 이상 징후를 평가한다.
5. 통합 검증 레이어: Cross-WF Jaccard, 상태 일관성 게이트, 보고서 품질 게이트로 중복, 보강 신호, 보고서 반영 여부를 검증한다.

평가: 5-Layer는 수집부터 보고서 반영까지 추적 가능한 파이프라인을 제공한다. 다만 문서상 실제 점수 계산식, 임계값, 모델 프롬프트, 예외 처리 코드가 복사본에 없으므로 재현성은 소스 확보 전까지 제한된다.

### 2.3 품질 게이트와 상태 검증

문서상 검증 체계는 SOT 체크 68개, 4계층 품질 방어, L2b QC 14개, PG2 체크 9개, 타임라인 QC 11개, 서사 게이트 5개, TERM 충실도 3개, CG 완결성 10개, 대시보드 체크 6개 등 총 145개 이상으로 요약된다. 또한 SCG는 raw data와 report 사이의 신호 수 일관성, 날짜 일치 등을 검사한다.

이 구조는 hallucination 방지와 산출물 누락 방지에 강하다. 반면 체크 수가 많아질수록 실패 원인 추적과 false positive 관리가 중요해진다. 체크 정의가 YAML/스크립트/문서에 분산되어 있다면 변경 영향 분석 도구가 필요하다.

## 3. 보고서 생성 파이프라인

### 3.1 EN/KO 이중언어 산출

README에는 `에이전트 (EN) → 산출물 (EN) → 번역 에이전트 → 산출물 (KO)` 흐름이 명시되어 있다. 예시 산출물은 `environmental-scan-YYYY-MM-DD.md`와 `environmental-scan-YYYY-MM-DD-ko.md`다. STEEPs 용어 보존 100%가 품질 목표로 제시되어 있으며, v3.6.0에서는 `title_ko` 존재 검증이 전체 4개 WF에 강제된다.

평가: 원문을 EN으로 먼저 생성하고 KO를 번역 산출물로 만드는 방식은 글로벌 소스 통합에는 유리하다. 다만 WF3처럼 한국어 원천이 핵심인 경우 한국어 원문 → EN 요약 → KO 재번역 구조가 되면 의미 손실이 생길 수 있다. WF별 원천 언어를 보존한 bilingual field 구조가 더 안전하다.

### 3.2 보고서 병합

개별 WF 보고서는 `report-merger.md` 계층에서 통합되는 것으로 문서화되어 있다. `integrated-report-skeleton.md`, `weekly-report-skeleton.md`, `lazy_report_generator.py`가 보고서 생성과 지연 생성을 담당하는 구성으로 보인다. 주간 명령 `/env-scan:run-weekly`는 신규 스캔 없이 메타 분석만 수행한다.

평가: 일간과 주간을 분리한 구조는 비용과 실행 시간을 절감한다. 지연 보고서 생성은 사용자 승인 또는 품질 게이트 이후 산출물을 만들 수 있어 효율적이지만, 입력 스냅샷 버전 고정이 없으면 같은 날짜 보고서가 재생성 시 달라질 수 있다.

### 3.3 HTML 대시보드

문서상 v3.6.0의 핵심 변경은 이중언어 대시보드, EN/KO 시그널 제목, 내러티브 서브탭, D3.js 인터랙티브 시그널 맵이다. 시그널 맵은 STEEPs x Impact 축을 사용한다. 대시보드 품질은 DB-001~DB-006 체크와 CG-010 대시보드 완결성 체크로 검증된다.

평가: D3.js 기반 대시보드는 신호 클러스터와 영향도 분포를 탐색하는 데 적합하다. 다만 HTML 산출물이 정적 파일이라면 대량 시그널에서 렌더링 성능과 브라우저 메모리 사용이 병목이 될 수 있다. 시그널 수가 증가하면 서버 사이드 집계, progressive rendering, 필터 인덱스가 필요하다.

### 3.4 승인 및 수정 흐름

명령 체계는 `/env-scan:review-filter`, `/env-scan:review-analysis`, `/env-scan:approve`, `/env-scan:revision`, `/env-scan:status`를 제공한다. 이는 자동 수집과 사람 승인 사이에 명시적 게이트를 둔 구조다. 코드 품질 관점에서 이 게이트는 운영 안정성에 유리하지만, 상태 파일 `workflow-status-latest.json`과 날짜별 상태 파일이 불일치할 때 어떤 파일이 우선인지 명확히 관리해야 한다.

## 4. 코드 품질 평가

### 4.1 구조

장점:
- WF1~WF4를 소스, 데이터 루트, 실행 명령, 검증 프로필 단위로 분리해 책임 경계가 선명하다.
- 설정 파일(`domains.yaml`, `sources*.yaml`, `workflow-registry.yaml`, `thresholds.yaml`) 중심 구조라 소스 추가와 정책 변경이 코드 수정 없이 가능하도록 설계되어 있다.
- SOT, SCG, VEV, Pipeline Gate 등 산출물 품질을 방어하는 계층이 명확하다.

우려:
- 복사본 기준 실제 Python/agent/command 소스가 누락되어 문서와 코드 구현의 일치성을 검증할 수 없다.
- 약어가 많고 게이트가 많아 신규 기여자가 전체 흐름을 이해하기 어렵다. `scanning_modules.txt` 같은 모듈 목록이 실제 import graph와 동기화되는지 자동 검증이 필요하다.
- 워크플로우 독립성은 좋지만 공통 로직이 과도하게 복제되면 WF별 버그 수정이 누락될 수 있다. 공통 수집 인터페이스와 공통 검증 라이브러리의 경계가 중요하다.

### 4.2 확장성

확장 포인트:
- 새 워크플로우는 `workflow-registry.yaml`에 ID, data_root, sources_config, validate_profile을 추가하는 방식으로 확장 가능해 보인다.
- 새 소스는 WF별 `sources*.yaml`에 추가하고, 도메인 분류는 `domains.yaml`에 키워드와 STEEPs 정의를 추가하는 구조다.
- 대시보드는 STEEPs x Impact 모델을 유지하면 WF가 늘어나도 공통 시각화에 편입 가능하다.

확장 리스크:
- WF가 늘어날수록 Cross-WF Jaccard 비교는 조합 수가 증가한다. 단순 pairwise 비교라면 O(n^2) 성능 문제가 생긴다.
- 다국어 소스 확장은 번역, 토큰화, 중복 제거, 날짜 정규화 비용을 선형 이상으로 증가시킨다.
- D3 정적 대시보드는 신호 수가 많아질 때 클라이언트 렌더링 병목 가능성이 있다.

### 4.3 테스트와 검증

강점:
- `validate_registry`, `validate_report`, `validate_report_quality`, `validate_phase2_output`, `validate_timeline_map`, `validate_timeline_map_quality`, `narrative_gate`, `validate_completion`, `validate_state_consistency` 등 검증 스크립트가 문서상 존재한다.
- 구조 검증, 기능 검증, 품질 검증을 분리한 VEV 사후 3-Layer 정책은 장애 대응 기준이 명확하다.
- 실패 처리 정책은 Layer 1 즉시 재실행, Layer 2 실패 항목 재실행, Layer 3 경고 및 사용자 알림으로 단계화되어 있다.

부족한 점:
- 복사본에 테스트 코드와 fixture가 없어 실제 자동 테스트 실행 가능성을 확인하지 못했다.
- 검증 스크립트가 산출물 검사에 치우쳐 있다면 단위 테스트, 계약 테스트, 크롤러 mock 테스트가 부족할 수 있다.
- 외부 소스 크롤링은 네트워크 변동성이 크므로 golden fixture와 recorded response 기반 회귀 테스트가 필요하다.

### 4.4 보안과 운영 리스크

- 크롤러 계층은 외부 URL, HTML, RSS, API 응답을 다루므로 SSRF, 악성 HTML, 과도한 리다이렉트, 대용량 응답 제한이 필요하다.
- HTML 대시보드가 원문 제목과 요약을 삽입한다면 XSS 방지를 위한 HTML escaping이 필수다. 특히 다국어 뉴스 제목과 사용자 수정 피드백이 그대로 들어갈 수 있다.
- 외부 API 키나 크롤링 인증 정보가 설정 파일에 들어간다면 비밀 관리와 로그 마스킹이 필요하다.
- 상태 파일과 보고서 파일을 날짜별로 누적하므로 동시 실행 시 파일 잠금, 원자적 쓰기, 중복 실행 방지 lock이 필요하다.

### 4.5 성능 리스크

- WF4 다국어 뉴스와 Cross-WF 비교는 데이터량 증가에 민감하다. 키워드 역색인, MinHash/LSH, 임베딩 캐시 같은 중복 탐지 최적화가 필요할 수 있다.
- D3 대시보드는 많은 노드와 링크를 한 번에 렌더링하면 느려진다. 필터링된 초기 뷰와 lazy loading이 권장된다.
- 번역과 LLM 기반 분석은 비용과 지연이 크다. 원문 hash 기반 캐시, 증분 처리, 실패 재시도 budget이 필요하다.
