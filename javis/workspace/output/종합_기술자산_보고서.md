### 1.2 공통점 (Cross-Project Commonalities)

#### A. 단계적 파이프라인 아키텍처
3개 프로젝트 모두 **다단계 순차 파이프라인**으로 설계되어 있다.

| 프로젝트 | 파이프라인 | 단계 수 |
|---------|----------|--------|
| 설교에디터 | 입력→기본틀→원고→슬롯→이미지→선택→DOCX→PPT | 9단계 |
| 글로벌뉴스 | 수집→전처리→피처추출→분석→집계→시계열→교차분석→신호→저장 | 8단계 (Stage 1~8) |
| 환경스캐닝 | 수집→정제→분류→신호평가→통합검증→보고서→대시보드 | 7단계 (5-Layer) |

#### B. JSON/YAML 중심 설정 구조
- **설교에디터**: `slots.json`이 파이프라인 전체의 Single Source of Truth
- **글로벌뉴스**: `state.yaml`이 중앙 SOT, 116개 사이트 메타는 어댑터별 관리
- **환경스캐닝**: `sources*.yaml`, `domains.yaml`, `thresholds.yaml` 등 설정 파일 구동

#### C. 품질 검증 게이트
모든 프로젝트가 **자동 검증 → 종료 코드 기반 게이트**를 구현한다.

| 프로젝트 | 검증 체계 | 검증 항목 수 |
|---------|----------|------------|
| 설교에디터 | validate_manuscript(28), validate_slots(4), 2-Agent 검수 | ~35개 |
| 글로벌뉴스 | 19개 결정론적 검증 스크립트 + D-7 동기화 + 5계층 QA | ~50+개 |
| 환경스캐닝 | SOT 68 + QC 14 + PG2 9 + 타임라인 11 + 서사 5 + CG 10 + DB 6 | **145+개** |

#### D. Streamlit/대시보드 시각화 계층
- 설교에디터: Streamlit 이미지 선택기 + HTML 미리보기
- 글로벌뉴스: Streamlit 6탭 대시보드
- 환경스캐닝: D3.js 인터랙티브 시그널 맵

#### E. 다국어/다문자 처리
- 설교에디터: 한글(Pretendard/나눔명조) + 헬라어 + 히브리어 (HarfBuzz RTL)
- 글로벌뉴스: 14개 언어 (Kiwi 한국어 + spaCy 영어 + SBERT 다국어)
- 환경스캐닝: EN/KO 이중언어 출력 + 11개 언어 소스

## 2. 프로젝트별 핵심 요약

### 2.1 My-Sermon-Editor

**한 줄 정의**: 성경 본문을 입력하면 설교 DNA에 맞춰 기본틀→원고→이미지→DOCX→PPT까지 자동 생성하는 AI 콘텐츠 파이프라인.

**아키텍처 핵심**:
- **슬롯 JSON 중심 설계**: `slots.json`이 파이프라인 전체의 SSoT. 이미지/텍스트 분기, 동기화, 검증의 모든 판단 기준.
- **Claude Code 스킬 오케스트레이션**: 9개 스킬이 자연어 라우팅으로 자동 호출. 전자동/단계별 2모드.
- **NLM 6벌 병렬 이미지**: 하나의 비주얼 브리프에서 6가지 스타일로 병렬 생성 → Streamlit 선택기로 최적 조합 선정.
- **멀티 포맷 출력**: 동일 원고에서 HTML 미리보기 + DOCX 인쇄본 + PPTX 슬라이드를 일관된 디자인 토큰으로 생성.

**기술 스택**: Python, Pillow, python-pptx, python-docx, Streamlit, uharfbuzz, NLM CLI

**스크립트 구조**: 핵심엔진 4 + 변환 3 + 슬롯동기화 4 + 이미지처리 6 + 아동부전용 3 + 후처리 3 + 스타일모듈 3 + 레거시 6 ≈ 34개

**재사용 패턴 7개**: 슬롯 기반 파이프라인, 다중스타일 병렬생성+선택, 양방향 동기화, 검증 게이트, 디자인 토큰 중앙관리, 파일 기반 비동기 이벤트, 부서분기+공용엔진

### 2.2 GlobalNews-Crawling-AgenticWorkflow

**한 줄 정의**: 116개 글로벌 뉴스 사이트에서 페이월을 우회하며 수집하고, 56개 NLP 기법으로 분석하여 5-Layer 신호 체계로 압축하는 Staged Monolith 빅데이터 분석 시스템.

**아키텍처 핵심**:
- **BaseSiteAdapter 패턴**: 450+ LOC 공통 모듈에서 URL 발견(RSS→Sitemap→DOM 3-Tier), 추출 체인, 에스컬레이션을 처리. 116개 어댑터는 CSS 선택자와 페이월 로직만 오버라이드.
- **7-Tier DynamicBypassEngine**: 딜레이 증가 → 쿠키 재설정 → Patchright 헤드리스 → 핑거프린트 우회 → CSS 4단계 적응 추출 → Never-Abandon → Claude 에스컬레이션.
- **56개 NLP + 5-Layer 신호**: Kiwi/spaCy 전처리 → SBERT 384d 임베딩 → KoBERT/RoBERTa 감성 → HDBSCAN 클러스터링 → STL/PELT 시계열 → Granger/PCMCI 인과 → 5-Layer 신호(L1 Fad ~ L5 Singularity 2-of-3 합의).
- **로컬 CPU 최적화**: Model2Vec 500x 가속, SQLite FTS5+vec 내장 검색, API 비용 $0.

**기술 스택**: Python, httpx, Patchright, SBERT, KoBERT, RoBERTa, HDBSCAN, UMAP, BERTopic, Model2Vec, STL, PELT, Prophet, Granger, PCMCI, DTW, SQLite, DuckDB, Parquet, Streamlit

**AgenticWorkflow DNA 유전**: 3단계 프로세스(R→P→I), 단일 SOT, 5계층 QA, P1 할루시네이션 봉쇄, Context Preservation, Safety Hooks

### 2.3 EnvironmentScan-system-main-v4

**한 줄 정의**: 4개 독립 워크플로우(일반/arXiv/네이버/다국어뉴스)로 약신호를 수집·분류·평가하고, STEEPs 프레임워크로 통합하여 EN/KO 이중언어 보고서와 D3.js 대시보드를 산출하는 전략적 환경스캐닝 시스템.

**아키텍처 핵심**:
- **4중 워크플로우 분리**: WF1(일반), WF2(arXiv), WF3(네이버), WF4(다국어뉴스). 각 WF는 상호 독립, 통합은 병합 계층에서만.
- **STEEPs 분류 + 5-Layer 약신호 처리**: 수집→정제→도메인분류→신호평가(FSSF/Three Horizons/Tipping Point/Anomaly)→통합검증.
- **145+ 품질 게이트**: SOT 68 + QC 14 + PG2 9 + 타임라인 11 + 서사 5 + CG 10 + DB 6. VEV 사후 3-Layer 검증(구조→기능→품질).
- **YAML 구동 설정**: 소스, 도메인, 임계값, 워크플로우 레지스트리 모두 YAML 파일로 관리. 코드 수정 없이 소스/정책 변경 가능.

**기술 스택**: Python, D3.js, YAML 설정 체계, LLM(분석/번역), 크롤링 모듈

**산출물**: EN/KO 이중언어 보고서, D3.js 인터랙티브 시그널 맵(STEEPs × Impact), 주간/일간 보고서

## 4. 크로스프로젝트 시너지 분석

### 4.1 시너지 매트릭스

```
                    설교에디터         글로벌뉴스         환경스캐닝
                 ┌──────────────┬──────────────┬──────────────┐
  설교에디터      │     ━━━      │   S1, S2     │   S3, S4     │
                 ├──────────────┼──────────────┼──────────────┤
  글로벌뉴스      │   S1, S2     │     ━━━      │   S5, S6     │
                 ├──────────────┼──────────────┼──────────────┤
  환경스캐닝      │   S3, S4     │   S5, S6     │     ━━━      │
                 └──────────────┴──────────────┴──────────────┘
```

### 4.2 시너지 상세

#### S1. 글로벌뉴스 크롤링 → 설교에디터 리서치 자동화
**현황**: 설교에디터는 설교 준비 시 시사 이슈, 배경 지식을 수동으로 검색.
**시너지**: 글로벌뉴스의 BaseSiteAdapter + NLP 분석을 활용하면, 설교 본문 관련 최신 뉴스/논문을 자동 수집하여 기본틀 작성 시 컨텍스트로 제공할 수 있다.
**구현 난이도**: 중 (어댑터 재사용 + 키워드 필터 추가)

#### S2. 설교에디터 PPT 엔진 → 글로벌뉴스 보고서 시각화
**현황**: 글로벌뉴스는 Streamlit 대시보드로만 시각화. PPT/DOCX 형식 보고서 없음.
**시너지**: 설교에디터의 `sermon_pptx.py` + `pptx_text_native.py` + `md_to_docx_v2.py`를 활용하면, 분석 결과를 경영진용 PPT/DOCX 보고서로 자동 변환할 수 있다.
**구현 난이도**: 중 (슬롯 JSON 포맷 어댑터 필요)

#### S3. 환경스캐닝 STEEPs + 설교에디터 설교 기획
**현황**: 설교 주제 선정은 신교수님의 영감과 시리즈 계획에 의존.
**시너지**: 환경스캐닝의 STEEPs 약신호 중 Spirituality/Social 카테고리 신호를 설교 기획 보조 데이터로 활용. "사회적 이슈와 성경적 메시지의 접점"을 데이터 기반으로 발굴.
**구현 난이도**: 낮 (보고서 참조 수준)

#### S4. 설교에디터 다국어 렌더링 → 환경스캐닝 다국어 보고서
**현황**: 환경스캐닝은 EN/KO 이중언어 텍스트 보고서만 산출.
**시너지**: 설교에디터의 HarfBuzz RTL + 다국어 폰트 렌더링 + PPT 네이티브 텍스트 기술을 활용하면, 환경스캐닝 보고서를 다국어 PPT/시각 자료로 확장할 수 있다.
**구현 난이도**: 중 (렌더링 엔진 추상화 필요)

#### S5. 글로벌뉴스 NLP → 환경스캐닝 분석 강화
**현황**: 환경스캐닝은 LLM 기반 분석 의존도 높음 (비용 발생).
**시너지**: 글로벌뉴스의 로컬 NLP 스택(SBERT/KoBERT/HDBSCAN/Model2Vec)을 환경스캐닝에 이식하면, LLM API 비용 절감 + 오프라인 분석 가능. 특히 Model2Vec 500x CPU 가속은 환경스캐닝의 단일 머신 제약에 직접 적용 가능.
**구현 난이도**: 높 (NLP 모델 통합 + 파이프라인 리팩터링)

#### S6. 환경스캐닝 워크플로우 분리 → 글로벌뉴스 확장
**현황**: 글로벌뉴스는 단일 파이프라인에 116개 사이트 통합.
**시너지**: 환경스캐닝의 WF1~WF4 독립 워크플로우 패턴을 글로벌뉴스에 적용하면, 지역별(A~J 그룹) 또는 언어별 워크플로우를 독립 실행하고 병합 계층에서 통합할 수 있다. 장애 격리 + 병렬 실행 효율 향상.
**구현 난이도**: 높 (아키텍처 리팩터링)

javis-common/
├── doc_gen/           ← md_to_docx_v2.py + docx_style/ (C3)

### 5.2 중기 (1~3개월)

#### 제안 4: 글로벌뉴스 NLP를 환경스캐닝에 이식 (S5)
- Model2Vec 500x 가속으로 환경스캐닝의 LLM API 비용 절감
- SBERT 다국어 임베딩으로 Cross-WF 유사도 비교 고도화
- SQLite FTS5+vec로 시그널 DB 즉시 검색

#### 제안 5: PPT/DOCX 보고서 자동 생성 (S2)
- 글로벌뉴스 분석 결과 → 슬롯 JSON 변환 → sermon_pptx.py로 경영진용 PPT

- 디자인 토큰을 프로젝트별로 분리하여 브랜딩 일관성 유지

글로벌뉴스가 상속받은 AgenticWorkflow 프레임워크 DNA를 다른 프로젝트에 적용:
- 3단계 프로세스(R→P→I) → 설교에디터 전자동 모드 강화
- Context Preservation → 세션 중단/재개 복원력
- Safety Hooks → 파괴적 명령 사전 차단
