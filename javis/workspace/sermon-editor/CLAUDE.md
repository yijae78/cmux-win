# My-Sermon-Editor

수요기도회 설교 원고 작성 프로젝트. 성경 본문을 입력하면 목사님의 설교 DNA에 맞춰 설교 원고를 작성한다.

## ⛔ 스킬 우선 호출 원칙

스킬이 존재하는 작업은 사전 파일 조사(Agent, Read 등) 없이 즉시 스킬을 호출한다.
스킬 내부에서 필요한 참조 파일을 알아서 로드하므로 중복 조사는 금지.

## ⭐ 이미지 생성 — 무조건 기존대로 NLM (2026-05-31 실험 후 신교수님 확정)

**확정 방침: 앞으로 이미지 작업은 무조건 기존대로 NLM에서 한다.** 평소엔 지금까지처럼 NLM 6벌 자동 생성·매핑·선택기까지 전부 자동 처리. 따로 묻지 말 것.

**Claude는 유료 모델(ChatGPT/Gemini) 직접그리기를 먼저 제안하지 않는다.** 신교수님이 필요할 때 **"이런 그림 그릴 프롬프트 알려줘"**라고 직접 요청하면 — 그때만 복붙용 프롬프트(16:9 사이즈 지시 + 글자 정확 표기 + 구성 고정 포함)를 설계해 준다. 신교수님이 구독 웹앱에서 직접 그려옴 → `images/ppt_NN.png`로 저장하면 나머지(매핑·미리보기·PPT)는 Claude 자동.

**⭐ 왜 NLM인가 (2026-05-31 PP10 σύμμορφος 5장 비교 실험 결론):**
"잘 그린 그림" ≠ "설교에 맞는 그림". 유료 모델이 디테일·해상도·표정·헬라어 정확도는 압도적이었으나, 신교수님은 1차·2차 모두 **NLM을 일관되게 선호**. NLM의 **절제·여백·신비·톤 일관성**이 설교 슬라이드에 더 낫다는 판정:
- 절제된 어둠+한 줄기 빛 → 메시지(원어 글자)로 시선 모음 (유료는 빛 화려해 분산)
- 흐릿한 그리스도 형상 = 신성한 거리감·초월 (또렷한 사실적 예수 얼굴은 부담)
- NLM 6벌 톤 일관 — 유료 한 장만 끼면 튐
- 작은 인물 = 하나님 앞 인간의 겸손·대비
→ **디테일·정확도로 이미지 추천하지 말 것.** 설교는 절제·신비·전체 톤이 우선.
→ 실험 자료 `sermons/20260603.../images/test*_PP10.png` 4장은 **삭제 금지** (향후 미감 판단 자료).

**API 직접 생성(Claude가 유료 호출)은 기본 금지.** OpenAI·Gemini 이미지 API는 무료 0·종량 과금(장당 50~240원). 신교수님이 "API로 그려줘"라고 **명시 요청할 때만** 실행. `GEMINI_API_KEY`는 등록됨(단 이미지 무료티어 0이라 유료). 상세: 메모리 `feedback_image_engine_policy`.

## ⭐⭐ 2026-05-30 본문 슬라이드 + 원고 미리보기 최종 표준 (모든 부서 공통, 「기울기」 정립)

> **이 섹션이 본문 슬라이드·미리보기·재생성의 최신 기준.** 아래 v2 섹션들의 수치(60pt 등)는 v3로 상향됨.
> 정본: `scripts/generate_text_slides.py` · `scripts/md_preview.py` · `scripts/image_selector.py` (전부 **전 부서 공용** — 한 번 고치면 장년부·새벽·금요·아동부 모두 적용)
> 의존성: `pip install -r requirements.txt` (uharfbuzz, freetype-py 추가). 새 PC(노트북·회사)도 설치 필요.

### 1) 본문 슬라이드 (generate_text_slides.py)
- **글씨 80pt 시작 / floor 56pt** (이전 60/40)
- **상단 헤더 밴드 기본 ON**: 진네이비 `(14,32,62)` + 황금 하단 엣지 `(251,191,36)` 6px + 흰 글씨. slots의 `band_color`/`band_gradient`/`band_gradient_dir`/`band_bottom_edge`/`header_band`로 부서별 override 가능. 밴드 제목 글씨는 본문과 동일 크기(폭 자동 축소).
- **하단 PP번호 푸터 제거**
- **highlight는 반드시 리스트** `["구절"]` (문자열이면 sorted()가 글자 단위로 쪼개 엉뚱한 글자 강조 — 방어코드 있으나 표준은 리스트). 강조 구절은 한 줄 안에 있어야 매칭됨.
- **히브리어 = HarfBuzz 셰이핑** (uharfbuzz + freetype-py): RTL 순서 + 니쿠드(모음 부호) 정확 배치. PIL raqm 미지원 대체. 히브리어 자동 감지, 설정 불필요.

### 2) 원고 미리보기 (md_preview.py)
- **[이미지]/[본문슬라이드] 설명**: 회색 이탤릭 `#94a3b8` + 이미지 **바로 아래 바짝**(직전 빈 줄 제거, 이미지 margin-bottom 0). 강조(황금)와 색 충돌 회피.
- **TOC·본문 PP 마커 = 2박스**: `[▶PP번호]` 진한 황금 배경(검정 글씨) + `제목` 연한 황금 `#fde68a`. 제목은 **slots.json의 desc**(「—」 앞 부분, 24자). 🖼 이미지 / 📖 본문 아이콘. **음표(🎵) 표기 제거**.
- TOC↔본문 **동일 제목**으로 어디서든 일관 인식.
- **앵커 `scroll-margin-top: 80px`** — sticky 줌바(약 55px)에 PP 마커가 가리지 않게. 목차 클릭 시 본문 ▶PP 마커가 정확히 보이는 위치로 이동.
- **반응형 (핸드폰/태블릿/PC 자동 적응)**: viewport + @media. 데스크탑=목차+본문 나란히 / 태블릿(≤1024px)=목차 250px 축소 / 모바일(≤768px)=목차 숨김 + 좌상단 **햄버거(☰) 오버레이** + 본문 전체폭 + 목차 항목 클릭 시 자동 닫힘. 핸드폰 핀치 줌 가능(단 전체화면 모드 제외).
- **전체화면 버튼 ⛶** (줌 바 안): **Fullscreen API**(진짜 전체화면, 주소창까지 숨김). 모바일 전체화면 시 핀치 줌이 막히는 것은 **수용**(신교수님 결정 2026-05-31 — 전체화면이 우선). UI 숨김 방식(핀치 유지하나 진짜 전체화면 안 됨)은 시도 후 **채택 안 함**.
- **사이드바 토글(◀/▶) 시 본문·토글버튼이 함께 좌우 이동**: 리사이저가 `content-main`의 `margin-left`와 `toc-toggle-btn`의 `left`에 **인라인 스타일**을 박으므로, `toggleSidebar()`에서 그 인라인을 직접 갱신해야 함(CSS만으론 인라인이 우선이라 안 움직임).

### 3) 이미지 재생성 워크플로우 — Claude가 처리 (봇 자동 금지)
- 재생성 버튼 → `_trigger.json` 생성 → 선택기에 **"🔄 Claude가 새 이미지를 만들고 있습니다…"** 표시 + 6초 자동 새로고침(`render_processing`).
- **auto_regen.py 봇 자동 NLM 호출 금지.** 신교수님이 "재생성" 한마디 → **Claude가 `_trigger.json`(슬롯+프롬프트) 감지 → NLM 재생성 → 슬롯 후보(`images/{prefix}_{slot}.png`) 교체 → `_trigger.json` 삭제 → 선택기 자동 새로고침**.
- 완료 시 `_selection.json`의 regenerate/prompts에서 해당 슬롯 제거. (prefix 매핑: A=A-WARM B=B-GLOW C=C-DRAMA D=A-EPIC E=B-FIRE F=C-TENDER)

### 4) 컨펌 → 미리보기 (이미지 포함)
- image_selector [🎉 최종 컨펌] → `status=confirmed` → `post_confirm.py`가 selected의 `ppt_{NN}.png`(선택 시 자동 복사됨)를 원고 ▶PP에 삽입 + md_preview로 preview.html 생성(base64 인라인) → 브라우저 오픈. **status≠confirmed면 post_confirm 중단**하므로 컨펌 버튼이 반드시 status를 confirmed로 바꿔야 이미지가 미리보기에 들어감.

## ⛔ 원어블럭 표준 포맷 (2026-05-03 신교수님 명시, 2026-05-12 보강)

원고에서 헬라어·히브리어 단어 분석은 **반드시** 6요소 BLOCK 형식, **두 곳**에 배치:

### 배치 위치 1: 기본틀 요약 아래 — 원어 모음
- 기본틀 요약 표 바로 아래, 본문(PP1) 시작 전
- 본문에서 다루는 **모든** 원어를 한곳에 모아 정리 (평문, `>` 없이)
```
(헬) 원어 (음역) — 어원 분석 = 의미 (절) — 추가 설명
개역개정: 「번역어」
```

### 배치 위치 2: 본문 각 위치 — 📖 원어블럭
- 원어가 등장하는 **모든** 서술 위치에 삽입 (서술형 슬롯 포함, 예외 없음)
```
**📖 원어블럭**

> (헬) **원어** (음역) — 의미 (절) — 추가 설명
> **개역개정**: 「번역어」
```

### 공통 규칙
**필수 6요소**: (히/헬) / 원어 / 음역 / 의미 / 절 / **개역개정 번역**.
**개역개정 번역 누락 절대 금지** (신교수님 명시).
**금지**: "원어 본문 (헬라어)" 섹션으로 본문 전체를 통째 박는 것.

### ⛔ 검증 체크리스트 (DOCX 변환 전 필수 실행)
1. 기본틀 요약 아래에 원어 모음 섹션 존재 확인
2. 본문 내 모든 원어 분석 지점에 📖 원어블럭 존재 확인
3. `grep -c "개역개정" 원고*.md` 결과가 원어 수의 **2배** (모음 1회 + 본문 1회)
4. **하나라도 누락 시 DOCX 변환 진행 금지**

    - [이미지N] → images/ 폴더의 실제 이미지 삽입
    - 찬양 악보 → 이미지 삽입
    ↓
[4] output/ 폴더에 DOCX 저장
```
