## 2. 관점별 추가 개선점 도출

### 📐 1. 아키텍처 관점 (Architecture)
* **단일 모놀리식 스크립트 결합도**:
  - `dashboard.py` 파일 내부에 Streamlit 테마, CSS/HTML 문자열 빌더, RPC 소켓 인터페이스, HTTPServer 구동, 시스템 메트릭 조회 로직이 단일 클래스/함수 수준으로 배치되어 있습니다. UI 프레임워크가 바뀔 경우 백엔드 모니터링 로직까지 동반 수정해야 하는 높은 결합도를 보입니다.
  - **개선안**: 데이터 수집 기능(소켓, API 파싱, 시스템 자원)을 독립 모듈(예: `collector.py`)로 격리하고, UI 파일은 단순히 렌더링에만 집중하도록 단방향 데이터 바인딩 아키텍처로 분리해야 합니다.
* **Streamlit 생명주기와 백그라운드 스레드의 불안정한 결합**:
  - `st.session_state`를 기반으로 백그라운드 스레드에서 HTTP 서버를 기동(`_start_data_server`)하는 방식은 Streamlit 세션 강제 리셋이나 가상 샌드박스 환경 재구동 시 포트 바인딩 중복 에러 혹은 스레드 생명주기 유실을 초래할 수 있습니다.
  - **개선안**: 데이터 서버 기동 로직을 Streamlit 내부가 아닌, 별도의 독립 프로세스나 서비스로 분리하여 실행할 것을 권장합니다.

### ⚡ 2. 성능 관점 (Performance)
* **동기식 디스크 I/O 블로킹 오버헤드**:
  - [get_token_usage](file:///C:/dev/cmux-win/javis/dashboard.py#L400) 및 [get_claude_contexts](file:///C:/dev/cmux-win/javis/dashboard.py#L334) 함수는 캐시가 만료되는 주기에 따라 사용자 홈 디렉토리 하위의 수많은 `.jsonl` 세션 로그 파일을 동기식으로 로드 및 루프 파싱합니다. 이 연산이 실행되는 동안 UI 반응성 및 데이터 요청 핸들러가 순간적으로 프리징(Block)될 수 있습니다.
  - **개선안**: 파일 스캐닝 및 파싱 태스크를 백그라운드 워커 스레드나 비동기(`asyncio`) 스레드 풀에서 주기적으로 돌며 메모리 캐시를 갱신하게 하고, 핸들러는 단순 O(1) 메모리 참조만 하도록 논블로킹(Non-blocking) 아키텍처로 전향해야 합니다.
* **통합 iframe 렌더링 비용**:
  - 깜박임을 줄이기 위해 components.html() 단일 iframe 내에 2000px 규모의 전체 HTML 문서를 매번 새로 구성하여 리페인팅하고 있습니다. 이는 클라이언트 웹 브라우저의 렌더링 엔진 부하를 급격히 올리는 요소입니다.
  - **개선안**: 전체 DOM 리렌더링 대신 데이터만 JSON API 형태로 주고받아 특정 상태 요소만 리페인팅하는 SPA(Single Page Application)나 React/Streamlit 양방향 컴포넌트 방식으로 전환해야 합니다.

### 🧪 3. 테스트 가능성 관점 (Testability)
* **강한 인프라 의존성 주입 부재**:
  - [gather_fleet_data](file:///C:/dev/cmux-win/javis/dashboard.py#L121) 내의 `_socket.socket`, [get_rate_limits](file:///C:/dev/cmux-win/javis/dashboard.py#L174)의 `http.client.HTTPSConnection` 등 소켓 및 네트워크 커넥션 인스턴스를 내부에서 하드코딩 형태로 직접 생성하고 있습니다.
  - 네트워크 단절 환경이나 CI/CD 파이프라인 상에서 외부 API 모킹(Mocking) 및 가상 테스트를 수행하기가 극도로 어렵습니다.
  - **개선안**: 네트워크 클라이언트와 파일 로더를 인터페이스화(Abstraction)하고, 생성자나 인수를 통해 해당 클라이언트를 주입받는 DI(Dependency Injection) 방식을 채택해야 유닛 테스트 작성이 쉬워집니다.

### 🔒 4. 보안 강화 관점 (Security)
* **의미 없는 비용 유발식 Rate Limit 헤더 조회**:
  - Anthropic API의 통합 사용량 헤더를 가져오기 위해, [get_rate_limits](file:///C:/dev/cmux-win/javis/dashboard.py#L174)에서 Haiku 모델 호출을 위한 더미 메시지 POST 요청을 지속적으로 날리고 있습니다.
  - API 키가 만료되거나 크레딧이 소진된 경우 레이트 리미트 통계 자체가 에러를 반환하며 중단될 뿐만 아니라, 15초 캐시 주기에 따라 지속적인 API 과금이 유발됩니다.
  - **개선안**: Anthropic에서 별도의 비용 유발 없는 단순 Metadata 조회/Usage 조회 엔드포인트를 열어주지 않는 한, 파일 스캔 기반의 예측 로직으로 우회하거나 헤더 획득 주기를 대폭 늘려(예: 5분~10분) 불필요한 과금과 트래픽을 예방해야 합니다.
* **데이터 포트의 무인증 개방**:
  - 깜박임 방지용 데이터 서버가 로컬 루프백(`127.0.0.1:8501`)으로 설정되어 있지만, 포트가 열린 상태에서 어떠한 권한 검증이나 인증 헤더 체크도 진행하지 않고 있습니다.
  - 같은 PC의 샌드박스형 브라우저 앱 또는 타 권한 프로세스가 해당 포트를 임의 조회하면 전체 Fleet의 동작 내역 및 민감한 로깅 데이터가 온전히 유출될 위험이 존재합니다.
  - **개선안**: 초기 렌더링 때 검증용 임의 토큰(One-Time Token 등)을 발행하여, JS Fetch 요청 시 헤더 검증을 수행하는 절차를 마련해야 합니다.
